# Fase 9.1 — Runtime cross-platform (paths/FS + degradación elegante por SO)

## Diagnóstico de supuestos Windows-only

Hallazgos principales en runtime:

1. `cashdrawer.ipc.ts` usaba PowerShell RAW (`powershell.exe`) para modo impresora sin guardas por plataforma.
2. `serialport` se importaba en top-level, con riesgo de fallo de carga nativa y crash temprano en sistemas donde el módulo no esté disponible en runtime.
3. UI POS sugería `COM3` como valor por defecto incluso fuera de Windows.

## Cambios aplicados

### 1) Cashdrawer robusto por plataforma

Archivo: `app/electron/ipc/cashdrawer.ipc.ts`

- `serialport` pasa a carga lazy (`import('serialport')`) bajo demanda.
- Si `serialport` no está disponible en runtime, se devuelve error controlado (`SERIALPORT_UNAVAILABLE`) en vez de crash al arrancar.
- `mode: printer` ahora valida plataforma:
  - Windows: comportamiento existente (PowerShell RAW).
  - macOS/Linux: respuesta de error explícita y segura (`UNSUPPORTED_PLATFORM`) sin crash.

### 2) Default de puerto serial en UI según SO

Archivo: `app/renderer/src/pages/POS.tsx`

- Windows: mantiene `COM3` como default.
- macOS/Linux: default operativo genérico `/dev/ttyUSB0`.

### 3) Installer/autodetect con comportamiento explícito no-Windows

Archivos: `app/electron/db/mysql.autodetect.ts`, `app/electron/ipc/installer.ipc.ts`

- En no-Windows, `autoDetectAndConfigureMySQL()` evita depender de señales NSIS y retorna `manual` cuando no hay config existente.
- En Windows se mantiene el flujo actual con señales NSIS (`install-type.json` / `mysql-install-info.json`) sin cambios.
- `installer` mantiene contrato IPC actual; no se toca packaging en esta fase.

### 4) Facturación/impresión con hardening de runtime

Archivos: `app/electron/invoice/invoicePdf.ts`, `app/electron/modules/sales/sales.service.ts`

- Nombre de archivo PDF sanitizado para evitar caracteres inválidos por SO.
- Flujo de impresión valida disponibilidad de impresoras y falla con error explícito (`NO_PRINTERS_AVAILABLE`) en lugar de error ambiguo.

## Compatibilidad por SO

### Windows x64

- Soporte completo esperado para flujo actual POS.
- Integraciones hardware (cajón por impresora RAW / serial) preservadas como flujo primario.
- Autodetect NSIS + setup conservan comportamiento actual.

### macOS x64

- App no debe crashear por cashdrawer printer-mode Windows-only.
- Modo impresora de cajón queda degradado con mensaje explícito.
- Modo serial depende de disponibilidad real de `serialport` en runtime.
- Autodetect se mantiene en `manual` cuando no hay config (sin depender de NSIS).
- Impresión puede requerir impresoras configuradas en SO; si no hay, se devuelve error controlado.

### Linux x64 (Debian/Ubuntu)

- Igual enfoque que macOS para degradación en printer-mode.
- Serial operativo si el módulo nativo está disponible y hay permisos de dispositivo.
- Autodetect en `manual` por defecto sin señales NSIS.
- Impresión depende de backend de impresión del SO y configuración local.

## Limitaciones pendientes

- Integración de cajón por impresora RAW sigue siendo Windows-first (intencional en esta fase).
- No se aborda todavía packaging/firma/distribución por OS (fuera de alcance 9.1).
- Dependencias nativas (`better-sqlite3`, `serialport`) requieren build/runtime matching por plataforma en release final.
- Setup/autodetect cross-platform queda funcional en modo manual; la experiencia “auto-instalada por NSIS” sigue siendo Windows-first.

## Validación recomendada

- Smoke test de arranque en macOS/Linux con canales IPC de cashdrawer para confirmar degradación sin crash.
- Smoke de operaciones core (login, ventas, productos, caja, backups) en modo SQLite/MySQL según disponibilidad.

## Cierre 9.1 (startup + flujos core runtime cross-platform)

Fecha de revisión: 2026-03-28.

Flujos core revisados específicamente:

1. **Arranque inicial (`runAppBootstrap`)**
   - Se validó registro de IPC temprano + dominio, inicialización DB, ventana principal y update-check sin cambios de contrato.
   - Se endureció el ciclo periódico de fallback-sync con `try/catch` para evitar errores ambiguos/no controlados durante el timer.
   - Se añadió log explícito de error controlado para diagnóstico operativo.

2. **Login/Auth (`auth:login`, `auth:logout`)**
   - Se verificó funcionamiento de sesión por `sender` en Main process y continuidad del contrato actual.
   - Sin supuestos Windows-only residuales detectados en este flujo.

3. **Productos (`products:*`)**
   - Se validó smoke de operación con contexto autenticado y autorización en Main process.
   - Sin supuestos Windows-only residuales detectados.

4. **Venta/Caja (`sales:*`, `cash:*`)**
   - Se validó smoke de operación autorizada en flujos representativos de venta y estado de caja.
   - Sin supuestos Windows-only residuales nuevos detectados en estos handlers core.

5. **Backups (`backup:*`)**
   - Se revisó export/restore/create-manual en IPC.
   - Se endureció `backup:create-manual` para dejar traza explícita en error (`[backup:create-manual] failed`) manteniendo retorno actual (`null`) y contrato renderer.

### Limitaciones por SO que permanecen (esperadas en 9.1)

- Integraciones hardware específicas (ej. cajón por RAW printer) siguen siendo Windows-first con degradación controlada en otros SO.
- Disponibilidad de impresión y dispositivos depende de configuración local del SO.
- Dependencias nativas requieren build/runtime matching por plataforma para release.
