# Fase 3 — Subtarea 3.4
## Pipeline de resincronización progresiva `fallback_operations` -> MySQL

## Diseño aplicado
Se implementó un pipeline incremental en `app/electron/db/fallbackSync.ts` con ejecución controlada desde el bootstrap (`app.bootstrap.ts`) cuando el resolver indica `mysql_primary`.

### Flujo general
1. Resolver del modo DB (`resolveDatabaseMode`) confirma `mysql_primary`.
2. Se asegura esquema local de `fallback_operations` con metadatos de sync:
   - `synced_at`
   - `sync_status`
   - `sync_attempts`
   - `last_attempt_at`
   - `last_error`
3. Se asegura tabla MySQL `fallback_sync_operations` para idempotencia por `op_id`.
4. Se toman pendientes (`synced = 0`) en orden estable `created_at ASC, id ASC`.
5. Se aplica operación por operación y se marca resultado local.
6. Si una operación falla, el ciclo se detiene para no desordenar replay.

## Operaciones soportadas en v1
- `sales:create`
- `cash:open`
- `cash:close`
- `expenses:add`
- `audit:log`

## Operaciones excluidas en v1
- Cualquier operación no listada arriba queda como `unsupported` y se marca localmente como resuelta para no bloquear la cola.
- En particular, esta subtarea **no** agrega sync de devoluciones ni de backup file-level.

## Estrategia de idempotencia y orden
### Identificador único
- Se usa `fallback_operations.id` como `op_id` estable de replay.
- `op_id` se persiste en MySQL (`fallback_sync_operations`) al completar aplicación.

### Orden
- Orden estricto por `created_at` y `id`.
- En error, se corta el ciclo en ese punto para preservar secuencia.

### Idempotencia razonable
- Antes de aplicar, se consulta `fallback_sync_operations` por `op_id`.
- Si ya existe, se considera `duplicate` y se marca localmente como sincronizada.
- Para reducir duplicación por reintentos, las operaciones sincronizadas ahora incluyen payload enriquecido (`schemaVersion: 2`, `input` completo) y IDs estables cuando aplica.

## Criterio de éxito
Una operación se considera exitosa cuando:
1. La operación de dominio se aplica en MySQL (o se detecta duplicada por clave), y
2. Se registra/actualiza `fallback_sync_operations(op_id, operation, status='applied')`, y
3. En SQLite se marca `synced=1` con `sync_status` (`synced` o `duplicate`).

## Manejo de errores y reintentos
- Si falla una operación:
  - `sync_status='failed'`
  - `sync_attempts += 1`
  - `last_error` y `last_attempt_at` se actualizan
  - el ciclo se detiene para no reordenar operaciones dependientes
- Reintento: se ejecuta en el siguiente ciclo automático cuando MySQL sigue disponible.

## Ejecución controlada
- Se ejecuta desde `runAppBootstrap()` cada 5 segundos.
- Solo intenta resincronizar si el modo efectivo actual es MySQL.
- Tiene lock en memoria (`syncInProgress`) para evitar solape de ciclos.

## Límites explícitos de esta versión
1. No hay worker persistente ni scheduler distribuido: el proceso corre en Main Process.
2. No hay resolución avanzada de conflictos de negocio (solo deduplicación básica y corte en error).
3. Operaciones legacy de fallback con payload incompleto quedan `unsupported`.
4. No se implementa todavía replay offline->MySQL para devoluciones ni reconciliación de stock avanzada.
