# Fase 9.2 — Packaging multiplataforma (Windows/macOS/Linux)

Fecha: 2026-03-28.

## Objetivo de esta ronda

Dejar una base de packaging incremental y segura para:
- Windows x64: NSIS (`.exe`)
- macOS x64: DMG (`.dmg`)
- Linux x64: AppImage + DEB (`.AppImage`, `.deb`)

Sin cambiar lógica funcional del POS ni contratos IPC/renderer.

## Diagnóstico inicial

Se detectaron estos bloqueantes de configuración:

1. Script de `package` orientado sólo a Windows.
2. `extraResources` apuntaba a un archivo fijo inexistente (`resources/mysql-8.0.msi`), lo que rompe packaging si no está presente.
3. `nsis.include` apuntaba a `build/installer.nsh` pero el archivo no existía en el repositorio.
4. `rebuild:native` sólo contemplaba `better-sqlite3`, faltando `serialport` para escenarios de packaging por plataforma.
5. No había chequeo rápido para validar que outputs/build resources clave existieran antes de empaquetar.

## Cambios aplicados

### 1) Scripts de packaging por plataforma

Se agregaron scripts explícitos:

- `package:prepare`
- `package:win`
- `package:mac`
- `package:linux`
- `package:linux:dir`
- `packaging:check`

### 2) electron-builder multi-target

Se amplió la configuración `build` para targets por SO:

- `win`: NSIS x64
- `mac`: DMG x64
- `linux`: AppImage + DEB x64

Además:
- Se mantiene `asar: true`
- Se agrega `asarUnpack: ["**/*.node"]` para módulos nativos
- `extraResources` pasa a directorio `resources` (no archivo fijo)

### 3) Recursos de packaging endurecidos

- Se crea `build/installer.nsh` mínimo para evitar fallo por include faltante.
- Se crea `resources/README.md` para documentar assets opcionales (por ejemplo MSI offline en Windows).

### 4) Dependencias nativas

`rebuild:native` ahora incluye:
- `better-sqlite3`
- `serialport`

Esto reduce el riesgo de empaquetar binarios nativos incompletos en build local por SO.

### 5) Validación base de packaging

Se agrega `scripts/packaging/check-packaging-config.cjs` para validar precondiciones mínimas:
- targets win/mac/linux presentes
- recursos base existentes
- outputs build (`dist/electron`, `dist/renderer`) presentes

## Estado por plataforma

### Windows x64

- Target configurado: NSIS (`.exe`)
- Referencia funcional principal del POS: **se mantiene**
- Puede incluir payload opcional de MSI en `resources/`

### macOS x64

- Target configurado: DMG (`.dmg`)
- No se resuelve notarización/firma en esta ronda (requiere credenciales externas)
- Runtime queda soportado con degradaciones ya documentadas para features Windows-first

### Linux x64 (Debian/Ubuntu)

- Targets configurados: AppImage + DEB
- Dependencias nativas dependen de build en entorno Linux compatible

## Bloqueantes y límites actuales

1. **Firma/notarización macOS**: pendiente por credenciales/certificados externos.
2. **Firma Windows (code signing)**: no incluida en esta ronda.
3. **Cross-build real de nativos** (`better-sqlite3`, `serialport`): recomendado construir en SO destino (o runners por SO).
4. **Updater por plataforma**: la estrategia de publicación/firmas por canal requiere cierre de release ops.

## Cómo construir manualmente (base)

1. Build base:
   - `npm run build`
2. Validación de packaging:
   - `npm run packaging:check`
3. Packaging por SO (en el SO destino):
   - Windows: `npm run package:win`
   - macOS: `npm run package:mac`
   - Linux: `npm run package:linux`

## Resultado de la subtarea

El proyecto queda preparado para iniciar empaquetado real por plataforma con configuración base consistente y menor riesgo de fallos triviales de recursos/scripts.

No se toca comportamiento funcional del POS ni contratos IPC/renderer.
