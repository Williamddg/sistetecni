# Fase 3.1 — DatabaseModeResolver

## Objetivo
Centralizar la decisión de motor de base de datos en un único componente para evitar reglas divergentes entre módulos.

## Fuente única de verdad
Se agrega `app/electron/db/databaseModeResolver.ts` con tres estados explícitos:

- `sqlite_local`
- `mysql_primary`
- `mysql_unavailable_fallback_sqlite`

La resolución efectiva pública sigue siendo compatible (`'sqlite' | 'mysql'`) mediante:

- `resolveEffectiveDbMode()`
- `isMySqlEnabledResolved()`

## Reglas de resolución
1. Si `config.json` no tiene `dbMode=mysql`, se usa `sqlite_local`.
2. Si `dbMode=mysql` pero falta configuración mínima MySQL, se usa `mysql_unavailable_fallback_sqlite`.
3. Si hay configuración completa, se prueba conexión MySQL:
   - éxito: `mysql_primary`
   - fallo: `mysql_unavailable_fallback_sqlite`

Incluye caché de 3 segundos para evitar pruebas de conexión repetitivas.

## Migración de consumidores críticos
Se actualizan consumidores para depender del resolver:

- IPC: `auth.ipc.ts`
- Módulo: `users.service.ts`
- Repos: `sales.repo.ts`, `cash.repo.ts`, `expenses.repo.ts`, `audit.repo.ts`
- `db.ts` mantiene `getDbMode()` por compatibilidad, delegando al resolver.

## Compatibilidad
- No se cambian contratos IPC.
- No cambia la firma pública de `getDbMode()`.
- `dbRouter.ts` conserva helpers (`isMySqlEnabled`, `getDbMode`) pero ahora son asíncronos y delegan al resolver.
