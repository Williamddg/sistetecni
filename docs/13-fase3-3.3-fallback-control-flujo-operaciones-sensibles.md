# Fase 3.3 — Endurecimiento de fallback y control de flujo (sin sync completa)

## 1) Operaciones revisadas y comportamiento ante caída de MySQL

Se revisaron operaciones sensibles en ventas, devoluciones, caja, gastos, auditoría y backups.

### Con política explícita vía `fallbackControl`
- `sales:create`
- `sales:return`
- `cash:open`
- `cash:close`
- `expenses:add`
- `audit:log`
- `backup:create`

Cuando MySQL está caído pero el modo deseado es MySQL (`mysql_unavailable_fallback_sqlite`),
las operaciones permitidas se ejecutan en SQLite de forma coherente; las no soportadas se bloquean explícitamente.

## 2) Endurecimientos aplicados

1. Se agrega `app/electron/db/fallbackControl.ts`:
   - Resuelve plan de ejecución por operación sensible (`mysql` o `sqlite`) usando `DatabaseModeResolver`.
   - Centraliza política por operación (`requiresMySql`, `trackWhenFallback`).
   - Bloquea explícitamente operaciones que requieren MySQL activo.

2. Se integra ese control en repos/servicios críticos:
   - `sales.repo.ts` (`sales:create`)
   - `sales.service.ts` (`sales:return`)
   - `cash.repo.ts` (`cash:open`, `cash:close`)
   - `expenses.repo.ts` (`expenses:add`)
   - `audit.repo.ts` (`audit:log`)
   - `backups.ipc.ts` (`backup:create` manual)

3. Se elimina ambigüedad de flujo “intentar manualmente MySQL/SQLite” en esos puntos,
   delegando decisión al plan resolver-backed.

## 3) Operaciones habilitadas/deshabilitadas en fallback

### Habilitadas en fallback SQLite (cuando MySQL no está disponible)
- `sales:create`
- `cash:open`
- `cash:close`
- `expenses:add`
- `audit:log`
- `backup:create` (manual)

### Deshabilitadas en fallback (bloqueadas explícitamente)
- `sales:return` (requiere MySQL activo)

## 4) Base mínima preparada para sync posterior (sin replay todavía)

Se crea registro local de operaciones ejecutadas en fallback con tabla SQLite:

- `fallback_operations`
  - `id`
  - `operation`
  - `payload_json`
  - `resolution`
  - `created_at`
  - `synced` (default 0)

Se puebla en best-effort mediante `markFallbackOperation(...)` **solo** cuando:
- la resolución fue `mysql_unavailable_fallback_sqlite`, y
- la política de la operación indica tracking.

No se implementa en esta fase:
- replay/sincronización,
- resolución de conflictos,
- colas/event sourcing complejos.

## 5) Límites y decisiones diferidas

1. Backup/restore de archivos sigue siendo flujo SQLite por diseño actual.
2. No se altera contrato IPC ni shapes de payload/respuesta.
3. No se fuerza fallback ante errores transitorios ocurridos a mitad de operación MySQL; el endurecimiento se centra en decisión de entrada coherente por resolver.
