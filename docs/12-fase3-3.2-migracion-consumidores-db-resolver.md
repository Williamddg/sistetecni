# Fase 3.2 — Migración de consumidores al DatabaseModeResolver

## Consumidores paralelos detectados en esta ronda

1. `app/electron/ipc/auth.ipc.ts`
   - Hacía decisión propia (`isMySqlEnabled`) y llamaba `authUserMySql`/`authUser` en el handler.
2. `app/electron/modules/products/products.service.ts`
   - Auditaba con `logAudit` (SQLite directo), saltándose la ruta resolver-backed de `audit.repo`.
3. `app/electron/ipc/backups.ipc.ts`
   - Auditaba con `logAudit` (SQLite directo), también saltándose la ruta resolver-backed.
4. `app/electron/modules/users/users.service.ts`
   - El helper legado `hasMySqlConfig` quedó con naming engañoso después de 3.1 porque ya no validaba “config”, sino modo efectivo.

## Migraciones aplicadas

1. `auth.ipc.ts`
   - Se elimina branching local SQLite/MySQL en el IPC.
   - Se delega en `authUser(...)` (queries/auth.queries.ts), que ya resuelve modo por resolver.

2. `products.service.ts`
   - Se reemplaza `logAudit` directo por `logAuditRepo`.
   - El audit logging queda alineado con la misma decisión de modo DB que el resto de repos.

3. `backups.ipc.ts`
   - Se reemplaza `logAudit` directo por `logAuditRepo` para evento `BACKUP_CREATE`.
   - No se cambian contratos IPC, payloads ni shape de respuesta.

4. `users.service.ts`
   - Se introduce helper interno `shouldUseMySqlUsers()` para reflejar comportamiento real.
   - Se conserva compatibilidad temporal exportando `hasMySqlConfig` como alias (legado) al helper nuevo.

## Decisiones paralelas eliminadas / encapsuladas

- Eliminada decisión local en IPC de auth.
- Eliminados usos de audit SQLite directo en products/backups.
- Encapsulada ambigüedad de naming en users (`hasMySqlConfig` queda como alias compatibilidad, no como fuente de decisión semántica nueva).

## Riesgos de migración async revisados y mitigación

1. **Riesgo:** uso antiguo síncrono de helpers async (`isMySqlEnabled`, `hasMySqlConfig`).
   - **Mitigación:** verificación de usos y actualización a `await` en consumidores activos.

2. **Riesgo:** cambio de flujo en auth IPC.
   - **Mitigación:** se mantiene misma respuesta/error; solo se delega la decisión de modo a una ruta ya resolver-backed.

3. **Riesgo:** auditoría en flujos product/backup.
   - **Mitigación:** se mantiene payload de auditoría y manejo de errores best-effort; cambia solo el backend seleccionado por modo efectivo.

## Consumidores que se dejan para subtareas siguientes (por riesgo o alcance)

1. Flujos de backup/restore siguen orientados a archivo SQLite (`getDbPath`, copia `.db`).
   - No se modifica aún para MySQL/offline-sync por estar fuera de alcance de 3.2.

2. `db/mysql.ts`, `db/mysqlClient.ts`, `db/mysql.autodetect.ts`, `ipc/mysql.ipc.ts`, `ipc/installer.ipc.ts`
   - Mantienen lecturas de config MySQL porque su propósito es instalación/conectividad/configuración, no selección de backend por operación de negocio.

3. `returnSaleByMode` en `modules/sales/sales.service.ts`
   - Conserva restricción actual (“solo MySQL”) sin ampliar comportamiento para SQLite en esta fase.
