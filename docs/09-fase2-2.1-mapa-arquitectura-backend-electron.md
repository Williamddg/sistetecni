# Fase 2 — Subtarea 2.1
## Mapa de arquitectura backend actual + plan de modularización incremental (sin refactor)

Fecha: 2026-03-27
Alcance: diagnóstico estructural de `app/electron` sin mover archivos ni cambiar lógica.

## 1) Mapa actual de arquitectura de `app/electron`

### Vista de capas actuales (estado real)

- **Entrypoint / Orquestación**
  - `main.ts`: crea ventana, registra IPC, licencia, updater, backup diario y boot de DB.
- **Puente de renderer a main**
  - `preload.ts`: expone API completa `window.api.*` con canales IPC.
- **Capa IPC (adaptadores de entrada)**
  - `ipc/*.ts`: handlers por dominio (auth, products, sales, cash, reports, users, audit, config, mysql, installer, autodetect, backups, cashdrawer).
- **Capa DB/repositorios/queries**
  - `db/*.ts`: acceso SQLite, selección de modo, repos híbridos, setup/installer y MySQL config.
  - `db/mysql/*.ts`: repos/queries específicas MySQL.
- **Servicios transversales**
  - `license/license.ts`
  - `invoice/invoicePdf.ts`
- **Tipos auxiliares**
  - `types/better-sqlite3.d.ts`, `db/audit.types.ts`

## 2) Inventario de archivos y responsabilidades

### 2.1 Bootstrap / runtime
- `app/electron/main.ts`
  - Registra todos los IPC, hace init de DB, backups y updater en un único archivo.
  - También define handlers de licencia y updater.

### 2.2 Bridge renderer-main
- `app/electron/preload.ts`
  - Expone gran superficie IPC en un solo objeto global (`api`) con varios `any/unknown`.

### 2.3 IPC por dominio
- `ipc/auth.ipc.ts`: login y decisión SQLite/MySQL (actualmente por `isMySqlEnabled`).
- `ipc/products.ipc.ts`: operaciones de inventario/pos.
- `ipc/sales.ipc.ts`: creación venta, suspendidas, detalle, return, impresión.
- `ipc/cash.ipc.ts`: apertura/cierre de caja + backups diarios en cierre.
- `ipc/expenses.ipc.ts`: gastos.
- `ipc/reports.ipc.ts`: reportes.
- `ipc/users.ipc.ts`: usuarios + cambio/reset password + auditoría.
- `ipc/audit.ipc.ts`: consulta auditoría.
- `ipc/config.ipc.ts`: lectura/escritura config + relanzamiento.
- `ipc/mysql.ipc.ts`: configuración/test/schema MySQL.
- `ipc/installer.ipc.ts`: wizard instalador DB.
- `ipc/autodetect.ipc.ts`: autodetección de escenario MySQL.
- `ipc/backups.ipc.ts`: backup/export/restore.
- `ipc/cashdrawer.ipc.ts`: puertos/impresoras/apertura cajón.
- `ipc/rbac.ts`: validación por permisos desde payload.

### 2.4 DB, repos y utilidades
- `db/db.ts`: SQLite singleton + migraciones + `getDbMode` con test de conectividad.
- `db/dbRouter.ts`: `isMySqlEnabled` por config estática.
- `db/queries.ts` (monolito): auth, users, products, sales, cash, expenses, reports, audit.
- `db/*repo.ts`: fachadas híbridas SQLite/MySQL por dominio (sales/cash/expenses/audit/reports/products).
- `db/mysqlConfig.ts`, `db/mysql.ts`, `db/mysqlClient.ts`: config, pool y utilidades MySQL.
- `db/dbInstaller.ts`, `db/mysql.autodetect.ts`: setup/autodetect.
- `db/migrations.ts`: esquema SQLite y compatibilidad.
- `db/mysql/*.ts`: implementación MySQL por dominio.

### 2.5 Servicios transversales
- `license/license.ts`: machine id, licencia local, validación online (Supabase).
- `invoice/invoicePdf.ts`: generación de PDF factura.

## 3) Dependencias/acoplamientos detectados

### Acoplamiento fuerte en bootstrap
- `main.ts` importa y registra casi todos los módulos de la app directamente.

### Mezcla de estrategias de selección DB
- Existen decisiones paralelas:
  - `getDbMode()` (conectividad real y fallback)
  - `isMySqlEnabled()` (flag de config)
  - `hasMySqlConfig()` local en `users.ipc.ts`.

### Mezcla de responsabilidades en IPC
- Varias capas IPC contienen lógica de enrutamiento DB y auditoría en lugar de delegar a servicios de aplicación.

### Monolitos y high fan-in
- `db/queries.ts` concentra demasiadas responsabilidades y es dependencia de múltiples IPC/repos.
- `db/migrations.ts` también es grande y mezcla migraciones históricas + compatibilidad.

### Dependencias transversales con riesgo
- `ipc/cash.ipc.ts` depende de `ipc/backups.ipc.ts` (`createBackup`), acoplando módulos de entrada.
- `ipc/rbac.ts` depende del role enviado en payload del renderer.

## 6) Plan incremental de modularización (subtareas sugeridas)

### 2.2 — Preparación de bootstrap sin mover dominio
- Extraer de `main.ts` únicamente funciones de arranque (`registerAllIpc`, `setupAutoUpdates`, `setupLifecycle`) a `bootstrap/*` manteniendo imports actuales.
- Criterio: cero cambio funcional.

### 2.3 — Tipar contrato de preload por dominios
- Separar exposición API en funciones por dominio sin cambiar canales IPC.
- Mantener compatibilidad completa con renderer.

### 2.4 — Fraccionar `queries.ts` por dominios internos
- Crear módulos `queries/*.ts` (auth/users/products/sales/cash/expenses/reports/audit).
- Dejar un `index` compatible para migración gradual.

### 2.5 — Delgazar IPC: delegar lógica a servicios de aplicación
- En cada `ipc/*.ts`, conservar validación de entrada + autorización + llamada a servicio.
- Evitar decisiones de persistencia dentro del handler.

### 2.6 — Unificar resolución de modo DB (preparación Fase 3)
- Introducir interfaz única de resolución (sin cambiar comportamiento) y adaptar módulos gradualmente.

## 7) Riesgos especiales por componente (pre-modularización)

- **main.ts**: punto único de fallo por concentración de bootstrap + handlers globales.
- **preload.ts**: superficie amplia de IPC y typing laxo.
- **ipc/**: mezcla de permisos, routing DB, auditoría y lógica transaccional en algunos handlers.
- **queries.ts**: monolito crítico; cambios directos aquí tienen alto riesgo de regresión transversal.
- **Selección SQLite/MySQL**: estrategias múltiples pueden divergir entre módulos.
- **auditoría**: caminos mixtos SQLite/MySQL y llamadas directas desde handlers.
- **config**: cambios de config pueden disparar reload/relaunch desde IPC.
- **installer/setup**: validación de “DB instalada” simplificada y caminos paralelos de schema.
- **license**: dependencia externa (Supabase) combinada con estado local.
- **backups**: funcionalidad en módulo IPC y dependencia cruzada desde caja.
- **cashdrawer/devices**: integración hardware/OS-específica (serial + PowerShell/raw printer).

---

## 8) Conclusión de Subtarea 2.1

La arquitectura actual está suficientemente mapeada para iniciar modularización incremental con bajo riesgo.
El primer frente recomendado por seguridad operacional es **bootstrap + límites de handlers IPC + fragmentación gradual de queries**.
